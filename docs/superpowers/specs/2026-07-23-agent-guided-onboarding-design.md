# Agent 引导式开场设计

## 目标

让第一次使用 `study-loop` 的学生知道下一步该回答什么、为什么回答，以及回答后 Agent 会执行什么。引导必须降低认知负担，同时保留现有的事件源、状态派生和 CLI 铁律。

## 非目标

- 不修改学习状态计算、事件 Schema、FSRS 规则或 CLI 参数。
- 不把 Agent 改造成固定菜单式程序。
- 不在对话中直接宣布用户已经掌握知识点。
- 不一次性要求用户填写完整课程配置。

## 交互原则

### 1. 先判断是否需要提问

- 用户已经明确说明目标时，直接路由，不重复询问。
- 用户只说 `/study`、"继续"或"帮我学习"时，先运行 `next_step.py`，再进入一次开场分流。
- 每轮最多问一个问题；选项控制在 3–5 个。

### 2. 总分流问题

目标不明确时，使用下面的意图分流：

```text
你今天想做哪件事？
A. 继续今天的学习
B. 做题/刷题
C. 修复一道错题
D. 查看学习状态
E. 准备考试
```

提问后用一句话说明选择会触发哪个流程，不把 CLI 菜单直接丢给学生。

### 3. 按分支逐步追问

- 刷题：依次确认模式、题量、输出形态；如果用户已经提供其中一项，不再重复问。
- 新课程：先确认课程名称和课程目录，再引导初始化；知识点骨架后续按考纲逐步建立。
- 错题修复：先确认题目或错因，再进入错误假设、缺失前提、错因类型三步归因。
- 查看状态：直接生成 dashboard、错因表或单点证据，不额外询问无关信息。
- 考试准备：先读取考试日期和当前 next-best-step；缺少日期时只提醒如何补充。

### 4. 对新手解释系统状态

遇到以下状态时，给出“状态含义 + 下一步动作”：

- 没有课程：说明课程工作区需要 `course.yaml`，给出 `init_course.py` 示例。
- 没有 KC：说明知识点是系统安排复习的骨架，建议先从考纲注册。
- 没有题目：说明需要注册真题/课后题，或明确请求生成迁移题。
- 没有到期卡：说明当前没有紧急 FSRS 复习，不把它误报成“已经掌握”。

## 受影响文件

| 文件 | 改动 |
|---|---|
| `SKILL.md` | 新增引导协议、分流问题和状态解释规则，更新刷题分支为逐步提问。 |
| `docs/USAGE.md` | 新增 Agent 引导示例，解释用户会看到什么问题。 |
| `README.md` | 更新典型请求，展示“先分流、再执行”的使用方式。 |
| `README_EN.md` | 同步英文示例，保持双语首页结构一致。 |
| `CHANGELOG.md` | 在 `[Unreleased]` 记录本次 Agent 路由文档更新。 |
| `tests/test_docs.py` | 增加关键引导契约检查，防止后续删除单问题规则或铁律。 |

## 错误与边界

- 用户拒绝回答时，给出当前可执行的最小下一步，不循环追问。
- 用户回答含糊时，只请求缺失字段，不重新展示完整菜单。
- 脚本失败时，解释命令用途和错误原因；不伪造成功状态。
- 引导文本只能描述脚本会做什么，不能绕过 `validate_question.py`、`derive_state.py` 或事件写入规则。

## 验收标准

1. `/study` 在意图不明时只问一个总分流问题。
2. 用户明确说“按考纲出 5 道 HTML 题”时，不再询问模式、题量和形态。
3. 刷题信息缺失时按顺序逐项询问，而不是一次抛出三组选项。
4. README 中文/英文示例包含引导式请求，H2 章节数量保持一致。
5. 现有测试全部通过，新增文档契约测试通过。
6. 现有 CLI、状态 Schema 和课程数据文件不发生行为变化。
